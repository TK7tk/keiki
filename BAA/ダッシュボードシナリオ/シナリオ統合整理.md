## 1. 概要

### 1.1 ダッシュボード利用者
- **主要ユーザー**: CS（カスタマーサクセス）担当者
- **副次ユーザー**: ブランド担当者、営業担当者
- **対象外**: 運営チームのチーム長、経営陣（別途専用ダッシュボードが必要）

### 1.2 ダッシュボードの確認目的
1. **数値エラー検知**: 計測の異常、設定ミス、システム障害の早期発見
2. **施策効果測定**: A/Bテスト、クリエイティブ変更、シナリオ改善の結果確認
3. **施策立案**: ボトルネック特定、改善機会の発見、新規施策の検討

### 1.3 対象サービス
すべてWeb集客成果向上のためのChatbotサービス

| サービス名 | 主要機能 | 目標指標 |
|-----------|---------|---------|
| **BOTCHAN Engagement** | 離脱防止POP-UP→LINE誘導→診断コンテンツ→CV | POP-UP Imp Rate, LINE登録数, CVR |
| **BOTCHAN EFO** | Web申込フォームのチャット化（1問1答形式） | フォームCVR, 項目別通過率 |
| **BOTCHAN AI** | AIチャットによる問い合わせ対応→適切ページ誘導 | COM開始率, AI経由CVR |
| **BOTCHAN Keeper** | 解約防止チャットボット（ログイン→理由聴取→防止施策） | 解約防止率, 流入数 |

### 1.4 施策例（全サービス共通）
- POP-UP/Chat起動条件の最適化
- シナリオ/クリエイティブの変更
- Chat内UIの改善（ボタン、色、サイズ等）
- A/Bテスト実施

## 2. BOTCHAN シリーズ統合ダッシュボード

---

### 2.1 コンセプト・機能概要

- **目的**: サービス横断で「ビジネス貢献度」を評価し、ROI 最大化と追加施策立案を支援する。
- **主な利用者**
  - 営業：契約更新・クロスセル提案
  - CS：サービス横断の施策評価・ナレッジ活用

**主な機能イメージ**

- KPI 異常値ハイライト
  - 4 サービス全体の「総 CV 数」「総売上（MRR/Gross）」の目標に対する進捗度を大きく表示
  - 前日比・前週比で ±X%以上の異常変動を検知し、サービス名と指標（例: Engagement Imp Rate）をアラート表示

- サービス別 / クライアント別 ROI/CPA 比較
  - クライアントごとの「BOTCHAN 全体の貢献 CV 数」と「BOTCHAN 全体の月額費用（基本料＋従量課金）」から BOTCHAN 総合 ROI または総合 CPA を一覧化
  - 施策を実施したクライアント（A/B テスト実行中など）の ROI の増減を色分け表示し、成功/失敗施策の全体影響度を把握

- 成長ドライバー分析
  - 貢献度が高いサービス（例：Engagement 経由 CV が多い）や特定案件（例：EFO で CVR が急伸）の抽出
  - 「Engagement で CV した友だちが、後日 Keeper で解約防止対象になった割合」など、サービス連携 KPI のトレンドを表示し、相互作用による新規施策（例：Engagement のオファー調整）を検討

- 追加プロダクト Simulation
  - 例：EFO のみ導入中のクライアントに対し、Engagement を追加導入した場合の追加効果（CV 数増加、MRR 増加）をシミュレーション

- 各サービス毎パフォーマンス概要
  - 当月実績と前日比・前週比 ±X% を表示
  - EFO：PVCVR / CV 数
  - Engagement：PVCVR / CV 数
  - AI：Com 数 / Com 率（Com 数 / PV）/ CV 数
  - Keeper：流入数 / 解約数 / 解約防止数
  - 各カードの「詳細」リンクから各サービス個別ダッシュボードへ遷移可能

- 施策検討・改善ナレッジ共有ダッシュボード
  - 各サービス毎に「××日から××施策を実施中」といった施策一覧を表示
  - 一定の閾値を超えた施策のサマリ（成功施策）を参照可能

### 2.2 施策の結果確認（ROI/CPA・施策サマリ・連携 KPI）

#### シナリオ INT-1：総合 ROI 評価と契約更新提案（営業）

- **目的**: BOTCHAN シリーズ全体の ROI を定量的に示し、契約更新交渉を有利に進める
- **対象ビュー**: クライアント別 ROI/CPA 比較、サービス連携 KPI
- **想定利用者**: 営業（例：鈴木）
- **利用タイミング**: 契約更新前

**Given**

- 2025/12/20、営業の鈴木が、来月契約更新を迎えるクライアント（例：ピュアライフ）の現状評価を行っている。

**When**

- 「クライアント別 ROI/CPA 比較」リストで、
  - ピュアライフの BOTCHAN 総合 ROI が目標値を上回っていること
  - 特に Engagement 経由の貢献 CV が増加していること
  を確認する。

**Then**

- ROI を根拠に契約継続を確信する。
- さらに「Engagement で CV した友だちが、後日 Keeper で解約対象になっていないことの割合」のトレンドを確認する。
- 「エンゲージメントで獲得した良質な顧客層への価値」を付加価値として提案資料に組み込む。

---

#### シナリオ INT-2：サービス横断施策の結果確認とナレッジ化（CS）

- **目的**: 複数サービスを組み合わせた施策の相乗効果を把握し、ナレッジとして再利用する
- **対象ビュー**: 施策検討・改善ナレッジ共有ダッシュボード、各サービスの CVR / CPA 指標
- **想定利用者**: CS（例：田中）
- **利用タイミング**: 施策実行後の振り返り

**Given**

- CS の田中が、AI と EFO を併用しているクライアントのうち、AI チャットで CVR が改善傾向にあることを把握している。

**When**

- 「施策検討・改善ナレッジ共有ダッシュボード」で、
  - 「EFO：項目 UI 変更施策」
  - 「AI：クイックリプライ改善施策」
  の施策サマリを確認する。
- それぞれの施策が単体で CVR を向上させつつ、
  互いに連携して総合 CPA を大幅に改善していることを確認する。

**Then**

- この成功事例を抽出し、「成功施策タグ / テンプレート評価」セクションに登録する。
- 他の類似クライアントへの提案資料に利用できるようにする。

### 2.3 新規施策の検討（成長ドライバー分析・シミュレーション・ナレッジ活用）

#### シナリオ INT-3：クロスセル提案の期待効果シミュレーション（営業）

- **目的**: 追加プロダクト導入によるインクリメンタル効果を可視化し、クロスセル提案を強化する
- **対象ビュー**: 追加プロダクト Simulation
- **想定利用者**: 営業（例：鈴木）
- **利用タイミング**: クロスセル提案検討時

**Given**

- 営業の鈴木が、現在 EFO のみ導入し CVR が伸び悩んでいるクライアントに Engagement のクロスセルを提案したいと考えている。

**When**

- ダッシュボードの「追加プロダクト Simulation」機能で、
  - 現状：「EFO のみ」
  - シミュレーション：「EFO + Engagement」
  に切り替えた場合のシミュレーションを実行する。
- EFO の既存 PV 数と Engagement の全社平均 CTR/CVR を基に、
  **CV 数 +X 件、総 MRR +Y%** となる想定値を算出する。

**Then**

- シミュレーション結果（グラフと数値）を提案資料に自動出力する。
- 具体的な期待効果と導入費用対効果をクライアントに提示する。

---

#### シナリオ INT-4：VOC からのボトルネック特定とサービス横断施策立案（CS）

- **目的**: 統合 VOC データからボトルネックを特定し、複数サービスを連携させた施策を立案する
- **対象ビュー**: AI セクション（Com 数 / Com 率 / CV 数）、VOC・未回答データ統合分析、Engagement 指標
- **想定利用者**: CS（例：佐藤）
- **利用タイミング**: 大きな改善テーマを設定したいとき

**Given**

- CS の佐藤が、AI クライアントの CVR 改善策を検討するため、統合ダッシュボードを確認している。

**When**

- 「AI：Com 数 / Com 率 / CV 数」のセクションで、AI 経由 CVR が低迷していることを確認する。
- さらに「VOC・未回答データ統合分析」パネル（AI の問い合わせデータ貼り付け .csv など）で、
  「料金が高い」や「解約方法」に関する質問が、
  CV したユーザー / CV しなかったユーザーの両方で急増していることを発見する。

**Then**

- AI シナリオ序盤で料金に関する FAQ を改善する（AI 施策）。
- 同時に Engagement で離脱するユーザー層に対し、限定オファー POPUP を出す（Engagement 施策）というサービス横断施策を立案する。
- 立案した内容を「タスク管理表」に起票する。

---

#### シナリオ INT-5：成功ナレッジからの新規施策適用（CS）

- **目的**: 他クライアントの成功施策をテンプレートとして再利用し、施策検討コストを削減する
- **対象ビュー**: 施策検討・改善ナレッジ共有ダッシュボード
- **想定利用者**: CS（例：田中）
- **利用タイミング**: 新規施策案を素早く作りたいとき

**Given**

- CS の田中が、担当クライアント（Engagement）の Imp Rate を改善したいと考えている。

**When**

- 「施策検討・改善ナレッジ共有ダッシュボード」で、
  他クライアントの成功事例を「Imp Rate 改善」タグで検索する。
- 「クリエイティブ A/B テスト：青色から黄色に変更」で CTR +X% の大幅な改善を達成した事例を見つける。

**Then**

- その成功施策で利用されたクリエイティブテンプレートを呼び出す。
- 担当クライアントの色調に合わせてクリエイティブを調整する。
- Engagement の詳細ダッシュボードから、A/B テストの新規施策として即座に設定する。


② ダッシュボード設計の目的から見た「足りないシナリオ」の指摘
今回整理されたシナリオは「CS・営業が個別案件を見て、異常検知・施策評価・新規施策検討をする」ところまではかなり網羅できています。

一方で、「パフォーマンスダッシュボード」というプロダクトに落とし込む時に、まだシナリオとして書ききれていない・弱いと思われる観点を広めに挙げると、ざっくり以下のようなギャップがあります。

1. 「どの案件から見るか？」の優先度付け・ポートフォリオ視点

現状：

各サービスで「特定案件の異常に気づく」「特定案件の A/B テストを見る」シナリオはある。

しかし、「CS が 10〜30案件を持っている前提で、今朝どの案件から見るべきか」という“入り口”シナリオがない。

足りない例：

「今週、ROI 悪化 or CV 数急減 × MRR 大きい案件を優先的に並べる“アラートインボックス”」を起点に、

→ そのカードから各サービスダッシュボードに Drill-down するシナリオ。

結果：

「全案件をざっと眺める」前提になっており、

忙しい CS の“最初の 15 分で、どこを見るか”という行動にダッシュボードが寄り添えていない。

2. ブランド側担当者（クライアント）の利用シナリオがほぼない

前提には「ブランド担当者も利用者」とあるが、シナリオはほぼ 社内 CS/営業専用。

足りない例：

ブランド担当者が

月次レポート前に「今月 BOTCHAN が追加でどれだけ CV/売上を作ったか」を一目で確認する。

「どの施策がうまくいっていて、次にどの施策をやるべきか？」をレコメンドカードで見る。

結果：

ダッシュボードの UI/難易度が「プロの CS 前提」になりがちで、

クライアントが自分で触って価値を感じるストーリーが不足している。

3. 目標・予実管理と「今月着地」の視点が限定的

Engagement の一部で「当月着地想定 MRR」に触れられているが、

全サービス・全クライアントを通した「目標 vs 実績 vs 予測」のシナリオは存在しない。

足りない例：

「今月の BOTCHAN 起因 CV 目標に対して、

現在進捗 80%、このままだと着地予測 95% なので、どのサービスのどの施策を強化すると目標達成できそうか」を検討するシナリオ。

結果：

現在のシナリオは「過去の結果を評価する」寄りで、

「未来の着地をコントロールするためのダッシュボード」としての使い方が弱い。

4. セグメント／チャネル別の深掘り視点が限定的

一部に

AI: 時間帯・URL 別

Keeper: 流入経路別
があるものの、

広く見ると、以下のような切り口がまだシナリオ化されていない：

デバイス（SP / PC）

流入チャネル（広告媒体 / オーガニック / メルマガ 等）

キャンペーン別（クリエイティブ毎 / 出稿面毎）

足りない例：

「同じ Engagement でも、Instagram 広告経由のユーザーは CVR 高いが、リスティング経由は低い → EFO 側の UI 変更が効いている可能性 → 媒体別で施策を出し分ける」などのシナリオ。

結果：

Web マーケの現場で当たり前に見る軸（媒体、キャンペーン、デバイス）との接続が薄く、

“BOTCHAN が広告〜LP〜フォーム全体のどこに効いているか”が立体的に見えにくい。

5. キャンペーン / シーズンの比較・振り返りシナリオ

今あるシナリオは、基本「ある期間の施策」単体の Before/After 比較。

足りない例：

「去年のクリスマスキャンペーン vs 今年のクリスマスキャンペーンで、

BOTCHAN の貢献 CV 数・CPA・解約防止数がどう変わったか」を比較して、

来年に向けた改善ポイントを洗い出すシナリオ。

結果：

短期施策の評価はできるが、

「季節イベントや大型キャンペーンのナレッジを溜める」という文脈が弱い。

6. 実験ポートフォリオ（A/B テスト一覧）の管理視点

各サービスに A/B テストの「結果を見る」シナリオはあるが、

「今、どのクライアントで、どんな実験が走っていて、いつまでに結果が出るのか」という
実験ポートフォリオの俯瞰がない。

足りない例：

「今月実施中の A/B テスト一覧ダッシュボード」

ステータス（実施中/終了/勝ちパターン確定）

期待インパクト（CV +◯％想定）

実施サービス（Eng/EFO/AI/Keeper）

結果：

「攻め」側の運用（常に何かテストが走っている状態）のマネジメントにダッシュボードが使いにくい。

7. ワークフロー・タスク管理とのつながりがダッシュボード外に逃げている

多くのシナリオが

「タスクを起票する」「施策リストに追加する」
ところで終わっており、

ダッシュボード上には

「起票されたタスクのステータス」

「施策 → 実行 → 結果 → ナレッジ化」というライフサイクル
をモニタリングするシナリオがない。

足りない例：

「このクライアントの BOTCHAN 関連施策の“今の状態”（ToDo / Doing / Done と結果）」を 1 画面で見るシナリオ。

結果：

ダッシュボードが“気づき”だけを提供し、

実行・完了・学習は別ツール任せになっているため、

CS の運用体験が分断されやすい。

8. サービス連携 KPI の「入口側」シナリオが弱い

統合ダッシュボードには

Engagement → Keeper のような連携 KPI のイメージは書かれている。

ただしシナリオとしては

既に 4 プロダクト入っている or 2 プロダクト併用中の案件の評価・シミュレーションが中心。

足りない例：

「Engagement だけ導入している全クライアントの中から、

Keeper を入れたら“解約防止効果が高そうな”クライアントを自動抽出する」など、

次にどのプロダクトを入れるべきかのレコメンド視点のシナリオ。

結果：

連携 KPI は“振り返り”には使えるが、

“次のクロスセル対象を見つける”ための入口としては定義されていない。

9. 品質リスク / コンプライアンス観点（特に AI）のシナリオ
