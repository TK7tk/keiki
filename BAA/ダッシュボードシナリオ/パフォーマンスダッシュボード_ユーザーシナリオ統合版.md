# BOTCHANパフォーマンスダッシュボード - ユーザーシナリオ定義書（統合版）

## 1. 概要

### 1.1 ダッシュボード利用者
- **主要ユーザー**: CS（カスタマーサクセス）担当者
- **副次ユーザー**: ブランド担当者（クライアント側）、営業担当者
- **非対象**: 運営チームのチーム長、経営陣（別途専用ダッシュボードが必要）

### 1.2 ダッシュボードの確認目的
1. **数値エラー検知**: 計測タグ脱落、設定ミス、システム障害の早期発見
2. **施策効果測定**: A/Bテスト、クリエイティブ変更、シナリオ改善の結果確認
3. **施策立案**: ボトルネック特定、改善機会の発見、新規施策（クロスセル含む）の検討

### 1.3 対象サービス概要
すべてWeb集客成果向上のためのChatbotサービス。

| サービス名 | 主要機能 | 目標指標 |
|:---|:---|:---|
| **BOTCHAN Engagement** | 離脱防止POP-UP → LINE誘導 → 診断コンテンツ → CV | POP-UP Imp Rate, LINE登録数, CVR |
| **BOTCHAN EFO** | Web申込フォームのチャット化（1問1答形式） | フォームCVR, 項目別通過率 |
| **BOTCHAN AI** | AIチャットによる問い合わせ対応 → 適切ページ誘導 | COM開始率, AI経由CVR, BAD数 |
| **BOTCHAN Keeper** | 解約防止チャットボット（理由聴取 → 防止施策） | 解約防止率, 流入数, ログイン成功率 |

### 1.4 共通施策例
- POP-UP/Chat起動条件の最適化（タイミング、URL、セグメント）
- シナリオ/文言の変更
- クリエイティブ（バナー、ボタン等）の変更
- UI（色、サイズ、配置）の改善
- A/Bテストの実施

## 2. サービス別ユーザーシナリオ

### 2.1 BOTCHAN Engagement

#### シナリオ ENG-1：データ欠損・異常の発見（数値エラー検知）
- **目的**: 数値エラー（計測タグ脱落・条件設定ミスなど）を早期発見する
- **想定利用者**: CS担当者（複数クライアント管理）
- **利用タイミング**: 日次チェック

**Given**
- CS担当が全クライアントのEngagement日次データを閲覧している。
- 前日までは正常な数値が計測されていた。

**When**
- 特定案件のPOP-UP Imp Rate（Imp/PV）が前日比50%以上急落（例: 0.30 → 0.15）しているのを発見する。
- 同時にImp数、Click数も急減していることを確認する。

**Then**
- 計測タグ脱落または表示条件設定エラーと判断し、個別クライアントの詳細画面へ遷移。
- ログを確認し、必要に応じて設定修正またはエンジニアへエスカレーションする。

#### シナリオ ENG-2：クリエイティブ A/Bテスト結果の評価（効果測定）
- **目的**: A/Bテストの成果を評価し、本採用の可否を判断する
- **想定利用者**: CS担当者
- **利用タイミング**: テスト終了時（月次等）

**Given**
- 特定案件で既存クリエイティブ(A)と新規(B)のテストを1ヶ月間実施済み。

**When**
- ダッシュボードで「LINE登録数」「診断通過率」「LP遷移率」「CVR」を比較。
- Bパターンの優位性（例: CVRがA比で+20%改善）と統計的有意性を確認する。

**Then**
- Bパターンの本採用を決定。
- 改善効果を反映した「当月着地想定MRR」の更新を確認し、クライアント向け報告資料を自動生成する。

#### シナリオ ENG-3：ファネルボトルネックの特定と改善立案（施策立案）
- **目的**: 離脱ポイントを特定し、CVR改善施策を検討する
- **想定利用者**: CS担当者
- **利用タイミング**: 施策検討時、定例会前

**Given**
- 担当案件のCVRが停滞しており、他の優良案件との比較分析を行う。

**When**
- 「友だち追加後のシナリオ完了率」が優良案件平均（50%）に対し、当案件（30%）と著しく低いことを発見。
- 推移グラフで完了率の低下トレンドを確認する。

**Then**
- ボトルネックを「シナリオ中途離脱」と特定。
- 「質問数の削減」「中間クリエイティブの改善」等の施策を立案し、タスク管理リストへ追加する。

### 2.2 BOTCHAN EFO

#### シナリオ EFO-1：項目別通過率の異常低下検知（数値エラー検知）
- **目的**: 項目単位の異常を検知し、設定ミスやシステムエラーを反映させる
- **想定利用者**: CS担当者

**Given**
- EFOダッシュボードで項目別通過率を監視している。

**When**
- 「電話番号/メアド」項目の通過率が前日（80%）から当日（20%）へ急落しているのを発見。
- 入力試行回数は正常（ユーザー流入はある）であることを確認。

**Then**
- 入力規制の誤設定または外部システム連携（SMS認証等）の失敗と判断。
- 原因を特定し、必要に応じて「ワンクリックロールバック」機能で前日の設定に戻す。

#### シナリオ EFO-2：フォームUI/コンテンツ改善効果の測定（効果測定）
- **目的**: UI改善がCVRに与えた影響を把握し、横展開を検討する
- **想定利用者**: CS担当者

**Given**
- フォーム内に「院詳細情報」を追加するA/Bテストを実施。

**When**
- ダッシュボードで「BOT内CVR」および「LP全体のCVR」を比較。
- UI変更により「希望クリニック選択」ステップの通過率が向上し、全体CVRが1.14倍改善したことを確認。

**Then**
- 施策の成功を確認し、成功要因を分析。
- 類似の商材（医療系等）を持つ他クライアントへの横展開を計画する。

#### シナリオ EFO-3：入力ファネルの最適化ポイント検討（施策立案）
- **目的**: 入力負荷の高いステップを特定し、離脱を防止する
- **想定利用者**: CS担当者

**Given**
- 入力ファネルの全ステップデータを分析している。

**When**
- 「住所」ステップの離脱率が25%と突出しており、平均入力時間も他ステップの2倍かかっていることを発見。
- ユーザーが複数回入力に失敗しているログを確認する。

**Then**
- 「郵便番号からの自動補完」や「文言の簡略化」を改善案として起票。
- 改善によるインパクト（想定CV増加数）をシミュレーションし、開発優先度を決定する。

### 2.3 BOTCHAN AI

#### シナリオ AI-1：AI応答エラーの早期発見と対処（数値エラー検知）
- **目的**: 回答品質の低下や未知の質問への対応漏れを検知する
- **想定利用者**: CS担当者

**Given**
- 日次のKPI（COM開始率、CVR等）を監視している。

**When**
- ユーザーからの「BAD数（新規性高）」が急増し、COM開始率が低下しているのを発見。

**Then**
- 「VOCデータ分析」セクションへ遷移し、未回答テキスト（新商品情報等）や誤回答の内容を特定。
- AIマニュアル（FAQ）への回答追加・修正タスクを即座に起票する。

#### シナリオ AI-2：VOCに基づいた文言改善の効果測定（効果測定）
- **目的**: ユーザーの声を反映した改善施策の有効性を確認する
- **想定利用者**: CS担当者

**Given**
- クイックリプライの文言を「クーポンはこちら」から「初回50%OFF」へ変更。

**When**
- 施策前後で「クーポン関連の質問件数」と「当該質問経由のCVR」を比較。
- CVへの貢献度がトップ3に浮上したことを確認する。

**Then**
- 施策成功と判断。次にCVRが低い「返品方法」等の質問に対しても、同様の文言最適化を計画する。

#### シナリオ AI-3：時間帯・URL別ターゲティングの検討（施策立案）
- **目的**: 高効率なセグメントを発見し、露出を最適化する
- **想定利用者**: CS担当者

**Given**
- 「時間別・URL別レポート」で詳細分析を実施。

**When**
- 深夜帯（22時〜2時）は流入が少ないが、CVRが全体平均の1.5倍高いことを発見。
- 特定の商品詳細ページからのCVRが平均以下であることを発見。

**Then**
- 深夜帯限定の「今だけオファー」表示強化や、特定ページ専用の初期シナリオ作成を立案する。

### 2.4 BOTCHAN Keeper

#### シナリオ KEE-1：認証エラー・連携エラーの多発検知（数値エラー検知）
- **目的**: 外部システム連携（API等）の不具合を早期発見する
- **想定利用者**: CS担当者

**Given**
- 日次の「不可リスト」でログイン失敗件数を監視している。

**When**
- ログイン失敗率が通常（5〜10%）のところ、当日25%まで異常上昇しているのを発見。

**Then**
- ユーザーログを確認し、クライアント側APIのタイムアウトか、BOT側のロジック異常かを特定。
- クライアントのシステム担当へ即座に連絡し、修復を依頼する。

#### シナリオ KEE-2：解約理由別防止策の効果測定（効果測定）
- **目的**: 防止コミュニケーションの有効性を検証し、ナレッジ化する
- **想定利用者**: CS担当者

**Given**
- 「価格が高い」という理由に対し、「お届けサイクル変更」の提案を優先するようシナリオ変更。

**When**
- 変更前後で理由別の解約防止率を比較。
- サイクル変更提案による防止成功率が大幅に向上（32% → 48%）したことを確認。

**Then**
- 成功パターンを確定。他の理由（使用感の不満等）への応用を検討し、社内ナレッジDBへ登録する。

#### シナリオ KEE-3：ファネル中途離脱のボトルネック特定（施策立案）
- **目的**: 理由聴取前の離脱を減らし、防止チャンスを増やす
- **想定利用者**: CS担当者

**Given**
- Keeperのファネル全体データを分析している。

**When**
- 離脱の58%が「理由選択前」に集中しており、そのステップのメッセージが長すぎる（平均200文字）ことを発見。

**Then**
- メッセージを短縮するA/Bテストを立案. 視覚的なクリエイティブの追加を検討する。

#### シナリオ KEE-4：流入経路別パフォーマンスの最適化（施策立案）
- **目的**: 流入経路（マイページ、FAQ等）に応じた対応を設計する
- **想定利用者**: CS担当者

**Given**
- 流入経路別レポートを分析。

**When**
- 「マイページ」からの流入ユーザーは解約意思が強く、防止率が他経路より著しく低いことを確認。

**Then**
- マイページ流入者専用の「特別クーポン」提示シナリオを立案する。

## 3. 統合ダッシュボード シナリオ

### 3.1 サービス横断の異常検知
- **目的**: 全プロダクトの状態を俯瞰し、重大な変化を見逃さない

**Given**
- 統合ダッシュボードのトップ画面を表示。

**When**
- 「総CV数」が前日比15%低下のアラートが表示。
- 内訳ドリルダウンにより、Engagementのみが低下し、他サービスは正常であることを特定。

**Then**
- Engagement詳細ダッシュボードへワンクリックで遷移し、原因調査を開始する。

### 3.2 総合ROI評価と契約更新提案（営業・CS）
- **目的**: 複数プロダクト併用の相乗効果を定量化し、契約継続・拡大へ繋げる

**Given**
- 来月契約更新を迎えるクライアントの評価を実施。

**When**
- 「クライアント別ROI/CPA比較」で、目標ROI（120%）に対し実績（145%）を確認。
- 「Engagement経由の新規獲得ユーザー」が「Keeperで解約防止対象になりにくい」というサービス間連携KPIの良好なトレンドを確認。

**Then**
- ROIと顧客質の高さを根拠に契約継続を提案。提案資料を自動生成する。

### 3.3 追加プロダクト・シミュレーション（営業）
- **目的**: クロスセルの期待効果を可視化し、提案の説得力を高める

**Given**
- 現在EFOのみ利用中のクライアントへEngagementを提案したい。

**When**
- シミュレーターに「現状（EFO）」と「追加（Engagement）」を入力。
- 業界平均や類似案件のデータを基に、追加される想定CV数とMRR、投資対効果（ROI）を算出。

**Then**
- 算出結果をグラフ化して提案資料へ出力。商談アジェンダをセットする。

### 3.4 統合VOC分析からの横断施策（CS）
- **目的**: サービスを跨いだ課題を発見し、一貫した解決策を提供する

**Given**
- AIプロダクトのCVR改善を検討中。

**When**
- AIのVOC分析で「料金・解約」への懸念がCV未達成者に多いことを発見。
- 同時に、Engagementの離脱理由でも「価格不安」が上位であることを確認。

**Then**
- AIでの料金FAQ改善と、Engagementでの価格訴求POP-UP、Keeperでの価格対抗策を連動させた統合施策を立案する。

## 4. 設計上の不足要素と今後の拡張性 (Gap Analysis)

### 4.1 ポートフォリオ・優先順位管理の不足
- **現状**: 特定案件の深掘りは可能だが、CSが持つ数十案件の中で「今朝どこを見るべきか」の提示が弱い。
- **必要要素**: 異常値・重要度（MRR等）・契約更新時期を掛け合わせた「アラートインボックス」機能。

### 4.2 ブランド側（クライアント）視点の不足
- **現状**: シナリオが社内CS/営業向けに偏っている。
- **必要要素**: クライアントが自ら触って価値を実感できる「サマリビュー」、専門用語のヘルプパネル、レコメンドカード。

### 4.3 未来予測・予実管理の不足
- **現状**: 過去の結果評価が中心で、目標に対する進捗管理が弱い。
- **必要要素**: 目標CV数に対する現状進捗（%）と、月末の着地予測。目標未達時の推奨施策の提示。

### 4.4 セグメント・外部チャネル連携の不足
- **現状**: デバイス、広告媒体、キャンペーン別などのマーケティング軸が薄い。
- **必要要素**: GA連携や媒体別データの取り込み、SP/PC別の出し分け評価。

### 4.5 ワークフロー・運用サイクルの分断
- **現状**: ダッシュボードで気づきを得た後、別ツール（Slack, Jira等）で実行するフローが追跡されていない。
- **必要要素**: ダッシュボード内での施策ステータス管理（ToDo/Doing/Done）、施策後の結果自動紐付け。

### 4.6 リスク管理・コンプライアンスの不足
- **現状**: AIの誤案内やブランド毀損への監視が弱い。
- **必要要素**: 特定のNGワード検知、規約違反に繋がる回答の自動アラート。

### 4.7 システム・非機能要件
- **データ更新**: 異常値の自動検知（リアルタイム）、基本KPIは日次。
- **エクスポート**: 生データのCSVエクスポート、Googleスプレッドシートへの自動同期。
- **セキュリティ**: ユーザー別の閲覧権限管理、個人情報のマスキング。
